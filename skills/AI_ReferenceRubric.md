# AI Research Reliability & Evidence Rubric

## Purpose

This rubric defines a standardized framework for evaluating the reliability, authority, verifiability, and production-readiness of information gathered or generated by AI systems.

It is designed for:

* AI agents
* Research copilots
* Coding assistants
* Autonomous workflows
* Technical architecture analysis
* Knowledge synthesis systems
* Enterprise AI governance

The goal is to ensure that:

1. Sources are trustworthy
2. Claims are verifiable
3. Outputs are auditable
4. Hallucinations are minimized
5. Users can independently validate the result

---

# Core Principle

AI outputs are NOT trusted by default.

Every important claim must:

* Reference evidence
* Explain source quality
* Provide confidence scoring
* Show verification status
* Expose reasoning traceability

---

# Source Hierarchy (Priority Order)

## Tier S — Highest Trust

### Definition

Official, authoritative, primary sources.

### Examples

* RFCs
* Official documentation
* Standards organizations
* Academic papers (peer-reviewed)
* Government publications
* Source code repositories from maintainers
* Protocol specifications
* Maintainer comments/issues
* Security advisories
* Official SDK/API references

### Score Range

95–100

### Examples

* OpenAI official docs
* AWS docs
* React official documentation
* Ethereum Improvement Proposals (EIP)
* Sui Foundation docs
* NIST publications
* arXiv + published conference version
* GitHub repo from original maintainers

### Required Conditions

* Must be directly attributable
* Must be publicly verifiable
* Must not be outdated
* Must contain primary information

---

## Tier A — High Trust

### Definition

Strong secondary sources with expert credibility.

### Examples

* Well-known engineering blogs
* Conference talks
* Senior engineer writeups
* Verified architecture deep dives
* Reputable journals
* Vendor engineering blogs

### Score Range

80–94

### Examples

* Cloudflare engineering
* Stripe engineering
* Uber engineering
* Netflix tech blog
* Martin Fowler
* ThoughtWorks
* InfoQ

### Risk

May contain opinionated interpretations.

---

## Tier B — Medium Trust

### Definition

Community-driven or semi-verified educational content.

### Examples

* StackOverflow
* Reddit expert threads
* Medium articles
* Dev.to posts
* Tutorials
* YouTube explainers

### Score Range

60–79

### Conditions

Only acceptable when:

* Cross-validated
* Technically reproducible
* Supported by higher-tier evidence

### Risk

High hallucination propagation risk.

---

## Tier C — Low Trust

### Definition

Unverified or weakly attributable content.

### Examples

* Random blogs
* SEO articles
* AI-generated summaries
* Aggregator websites
* Uncited tutorials

### Score Range

30–59

### Conditions

Must NEVER be sole evidence source.

---

## Tier D — Unacceptable

### Definition

Unverifiable, anonymous, misleading, or contradictory sources.

### Examples

* Clickbait
* Spam sites
* Fabricated citations
* Unsourced claims
* AI-generated content without provenance

### Score Range

0–29

### Policy

Must be discarded entirely.

# Detailed Scoring Criteria

## 1. Source Authority (25)

### Questions

* Is the source official?
* Is the author identifiable?
* Is the organization reputable?
* Is the source primary?

### Scoring

| Score | Meaning                           |
| ----- | --------------------------------- |
| 22–25 | Official primary source           |
| 17–21 | Highly reputable secondary source |
| 10–16 | Community source                  |
| 0–9   | Weak/unknown authority            |

---

## 2. Verifiability (20)

### Questions

* Can claims be independently verified?
* Are citations included?
* Are references traceable?
* Is evidence reproducible?

### Scoring

| Score | Meaning              |
| ----- | -------------------- |
| 18–20 | Fully verifiable     |
| 12–17 | Mostly verifiable    |
| 6–11  | Partial verification |
| 0–5   | Unverifiable         |

---

## 3. Freshness (15)

### Questions

* Is the content current?
* Is the technology rapidly changing?
* Is the version explicitly stated?

### Scoring

| Score | Meaning                    |
| ----- | -------------------------- |
| 13–15 | Current and versioned      |
| 9–12  | Slightly outdated          |
| 4–8   | Old but partially relevant |
| 0–3   | Obsolete                   |

---

## 4. Cross-Source Consensus (15)

### Questions

* Do multiple reputable sources agree?
* Is the claim contradicted elsewhere?
* Is there ecosystem alignment?

### Scoring

| Score | Meaning            |
| ----- | ------------------ |
| 13–15 | Strong consensus   |
| 9–12  | Moderate consensus |
| 4–8   | Mixed opinions     |
| 0–3   | Contradictory      |

---

## 5. Technical Accuracy (15)

### Questions

* Is the explanation technically correct?
* Does it match official specs?
* Are there logical inconsistencies?

### Scoring

| Score | Meaning             |
| ----- | ------------------- |
| 13–15 | Technically precise |
| 9–12  | Mostly accurate     |
| 4–8   | Some inaccuracies   |
| 0–3   | Incorrect           |

---

## 6. Reproducibility (10)

### Questions

* Can results be reproduced?
* Are implementation details included?
* Are examples testable?

### Scoring

| Score | Meaning             |
| ----- | ------------------- |
| 9–10  | Fully reproducible  |
| 6–8   | Mostly reproducible |
| 3–5   | Partial             |
| 0–2   | Not reproducible    |

---

| ------ | ----- | ---------------------------- |

# Mandatory AI Output Format

Every serious AI-generated response should contain:

## 1. Executive Summary

Short conclusion.

---

## 2. Evidence Sources

| Source           | Tier | Reliability Score | Notes                |
| ---------------- | ---- | ----------------- | -------------------- |
| Official Docs    | S    | 98                | Primary source       |
| RFC              | S    | 96                | Specification        |
| Engineering Blog | A    | 85                | Secondary validation |

---

## 3. Cross-Validation Status

Example:

* Verified across 4 independent authoritative sources
* No major contradictions found
* API behavior confirmed in official SDK docs

---

## 4. Confidence Level

| Confidence | Meaning          |
| ---------- | ---------------- |
| High       | Strong evidence  |
| Medium     | Some uncertainty |
| Low        | Weak evidence    |

---

## 5. Known Risks / Uncertainties

AI must explicitly state:

* Unknowns
* Assumptions
* Version mismatches
* Missing evidence
* Experimental APIs
* Spec ambiguity

---

## 6. Verification Checklist

| Check                     | Status |
| ------------------------- | ------ |
| Official docs checked     | ✅      |
| Multiple sources compared | ✅      |
| Version validated         | ✅      |
| Deprecated APIs checked   | ✅      |
| Reproducibility confirmed | ✅      |

# Hard Rules for AI Systems

## AI MUST NOT

### 1. Invent Sources

Never fabricate:

* Papers
* URLs
* APIs
* Benchmarks
* RFCs
* Statistics

---

### 2. Present Assumptions as Facts

All assumptions must be labeled.

---

### 3. Use Single Weak Sources

Especially:

* Reddit only
* Medium only
* AI summaries only

---

### 4. Ignore Versioning

Technology advice without versions is unreliable.

---

### 5. Hide Uncertainty

Uncertainty must be visible.

# Hallucination Detection Signals

## High Risk Signals

* No citations
* Generic wording
* No version numbers
* No reproducible examples
* Vague benchmarks
* "Many developers say..."
* "It is generally known..."

---

# Gold Standard Response Characteristics

A trustworthy AI response should:

* Cite authoritative sources
* Explain confidence
* Expose uncertainty
* Be reproducible
* Be technically precise
* Be independently auditable
* Separate fact from interpretation

# Enterprise Recommendation

For production AI systems:

Reject responses when:

* Reliability score <80
* No authoritative sources
* No evidence traceability
* Contradictory evidence unresolved

Require:

* Structured evidence output
* Confidence scoring
* Source taxonomy
* Verification checklist

---

# Final Principle

The quality of an AI system is determined less by how confidently it answers,
and more by how transparently it proves its answers.
